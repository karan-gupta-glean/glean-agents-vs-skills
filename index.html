Agents vs. Skills

Key distinctions between Glean's two agentic primitives — and a decision framework for when to use each.

A simple guide to understanding when to use Agents vs. Skills in Glean — no technical background needed.

View as
Core Premise

Skills are reusable instruction sets that extend what the Glean Assistant can do. Agents package skills, tools, and knowledge together with enterprise-grade governance, scoping, identity, and operational controls — purpose-built for high-value production workflows.

Both are powerful on their own. Together, they let you automate anything from a simple formatting task to a fully governed, cross-system workflow — all grounded in your company's knowledge.
The Simple Version

Think of Skills as quick recipes — they teach Glean's Assistant how to do a specific task (like writing in your brand voice or formatting a report). Agents are like dedicated team members — they have their own access, work independently, and can handle entire workflows end-to-end.

Skills make the Assistant smarter. Agents handle whole jobs on their own. Together, they automate everything from simple tasks to complex business processes.

What Makes Agents Different

Core architectural capabilities that distinguish agents from skills
1 Independent Identity agent-only
What Agents Enable
  • Execute with their own service account identity, separate from individual users
  • Access to privileged tools that end-users should never have
  • Background execution without requiring a logged-in user
  • Audit trails that distinguish agent actions from human actions
Why This Matters

Some operations require elevated permissions that users should never possess directly. An IT Admin Agent can reset passwords or provision software because it has privileged access, not because the requesting user does. This separates capability from user permissions.

Example
User asks: "Reset my password"
IT Agent → Executes with IT service account credentials (user never has this permission)
Audit log → Shows "IT Agent" performed the action, not the user
🤖 Agent
Executes with its own service account identity and privileged access
✨ Skill
Always executes in the user's security context, inheriting their permissions only
Bottom line: When a workflow needs its own identity and elevated permissions, that's an agent.
1 Agents Have Their Own Access agent-only
What This Means
  • Agents can do things on behalf of your team that individual employees can't (or shouldn't) do themselves
  • They work in the background — no one needs to be logged in
  • Every action is tracked separately so you always know what the agent did vs. what a person did
Why This Matters

Think of an IT Help Desk Agent: an employee asks to reset their password, and the agent handles it securely — without giving the employee direct access to the password system. It's like having a trusted specialist on your team who handles sensitive tasks so your people don't have to.

Real-World Analogy
Skill = A recipe card that helps you cook the meal
Agent = A personal chef who cooks for you using their own kitchen access
Bottom line: When you need something done that requires special access or should run without someone watching, that's an agent.
2 Installability & External Surface Deployment agent-only
What Agents Enable
  • Deployment to external surfaces (Slack channels, ServiceNow, Teams)
  • Embedded experiences within third-party systems
  • Standalone invocation outside the Glean Assistant
Why This Matters

Agents can be installed where users already work. An HR Agent lives in the #hr-help Slack channel. An IT Agent is embedded in the ServiceNow portal. Skills are invoked capabilities within the Assistant — they don't have an independent presence in external systems.

Example
✔ IT Agent — Installed in #it-help Slack channel, responds 24/7
✘ IT Skill — Can only be invoked inside Glean Assistant
Bottom line: When you need AI to meet users where they already work — in Slack, Teams, or ServiceNow — that's an agent.
2 Agents Live Where Your Team Works agent-only
What This Means
  • Agents can be added to Slack channels, Teams, or ServiceNow — wherever your people already are
  • Employees don't need to open a separate app — help comes to them
  • Available 24/7, even outside business hours
Why This Matters

Your team shouldn't have to change how they work to get AI help. An HR Agent can sit right in your #hr-help Slack channel, answering benefits questions instantly. A skill, on the other hand, only works when someone opens the Glean Assistant directly.

Real-World Analogy
Skill = A reference book in the library — helpful, but you have to go find it
Agent = A colleague sitting in your team's Slack channel, ready to help instantly
Bottom line: When you want AI to be available right where your team already works — in Slack, Teams, or your help desk — that's an agent.
3 Scoped & Constrained Execution Environment agent-only
What Agents Enable
  • Constrained tool access — agents only use tools relevant to their domain
  • Knowledge boundaries — IT agents see only IT docs, not finance data
  • Explicit tool and knowledge manifests per agent
  • Guaranteed guardrails — the agent cannot break out of its scope
Why This Matters

General-purpose assistants have broad access, creating security and relevance problems. An IT Helpdesk Agent should never search through finance documents. Agents enforce these boundaries by design. Skills operate in the user's context with their full access.

🤖 Agent
Scoped to IT documentation, ticketing systems, password reset tools, software provisioning
✨ Skill
Could accidentally surface finance data when helping with an IT question, creating compliance risks
Bottom line: When a workflow must be locked down to specific data and tools with guaranteed guardrails, that's an agent.
3 Agents Only See What They Should agent-only
What This Means
  • Each agent is restricted to only the information and tools it needs for its job
  • An IT agent can't accidentally access finance, HR, or legal data
  • Built-in guardrails keep the agent focused and safe
Why This Matters

You wouldn't give an IT help desk employee access to your company's financial records. The same logic applies to AI. Agents are purpose-built with clear boundaries — an IT Agent only sees IT documentation and tools. This keeps sensitive information safe and answers more relevant.

🤖 Agent
Like a specialist — only has access to what's relevant to their role
✨ Skill
Like a generalist assistant — can see everything you can see, which may be too broad
Bottom line: When you need to ensure AI only accesses specific information and stays in its lane, that's an agent.

Enterprise-Grade Controls

Governance and lifecycle management across agents and skills
4 Governance & Permission Control platform-wide
What This Enables
  • Centralized control over who can create, distribute, and manage agents and skills
  • Approval workflows for deployment (compliance review, security sign-off, rollback)
  • Scoped access control for which users and groups can invoke specific agents or skills
Why This Matters

In enterprises, not everyone should create production workflows that touch sensitive systems. Governance ensures organizational control over who can create, distribute, and invoke both agents and skills — especially for workflows involving code execution or sensitive data access.

Example

Your marketing team publishes a "Brand Voice" skill that enforces tone and messaging guidelines. With governance, admins control which skills are active org-wide, enforce approval flows before a skill goes live, and ensure everyone uses the latest approved version.

Platform approach: Glean extends the same governance primitives across both agents and skills — admin roles, sharing controls, and approval flows — so enterprises get consistent controls across the entire platform.
4 Admins Stay in Control platform-wide
What This Means
  • Your IT or admin team decides who can create, share, and use agents and skills
  • New agents and skills go through an approval process before going live
  • You control which teams have access to which capabilities
Why This Matters

Just like you wouldn't let anyone install software on company computers without IT approval, you want oversight over AI automations. Governance means the right people review and approve agents and skills before they go live — keeping your organization safe and compliant.

Example

Your marketing team creates a "Brand Voice" skill so everyone writes in a consistent tone. An admin reviews and approves it, then publishes it to the whole company — ensuring everyone uses the same, approved version.

Platform approach: Glean gives admins the same controls over both agents and skills — approval workflows, sharing settings, and access rules — all in one place.
5 Packaging, Versioning & Testing platform-wide
What This Enables
  • Versioned, deployable artifacts (Agent v1.2)
  • Independent test suites and regression testing per agent
  • Rollback capabilities when new versions fail
  • Staged deployments (dev → staging → production)
Example
Sales Agent v2.0 changes CRM update logic:

✔ Test v2.0 against regression suite before rollout
✔ Deploy to 10% of users, monitor for errors
✔ Rollback to v1.9 if issues arise
Platform approach: Glean supports versioning, testing, and staged deployments for both agents and skills — enabling the same software development rigor you apply to production code.
5 Safe Updates & Easy Rollback platform-wide
What This Means
  • Agents and skills are versioned — you always know which version is live
  • Updates can be tested with a small group first before rolling out to everyone
  • If something goes wrong, you can instantly revert to the previous version
Example
Your Sales Agent gets an update to how it logs deals:

✔ Test the update with a small pilot group first
✔ Roll it out to the full sales team once validated
✔ Roll back instantly if anything looks off
Platform approach: Glean lets you test, roll out gradually, and roll back both agents and skills — so updates are safe, not scary.

Decision Framework

Agents and skills are complementary — agents are production workflows that often use skills as their tools
Build This When Characteristics
Agent Needs governance, scoping, identity, or operational controls Independent identity, scoped tools/knowledge, governance, versioning, observability, SME ownership, cost optimization
Subagent Needs domain reasoning, no independent lifecycle Domain expertise, multi-step decisions, delegated execution within another agent
Global Skill Simple, reusable capability for many users; deployed by admin or builder Single-purpose, stateless, horizontal use case (brand tone, JIRA formatting) where standardization matters org-wide
Agent-Scoped Skill Specialized capability for one agent's domain Complex, domain-specific logic needed only by that agent
Personal Skill Built or imported by a user for personal use only Same as global skill, but idiosyncratic to the end user's preference (feedback giving style)
Use This When You Need To... Think of It Like...
Agent Automate an entire workflow that runs independently with its own access A dedicated team member who handles a specific job — like an IT help desk agent or a sales ops coordinator
Subagent Have an agent call on a specialist for a specific part of its job A specialist that your main agent consults — like an agent asking a compliance expert to review a step
Global Skill Give everyone a simple, reusable capability A company-wide template — like a "write in brand voice" or "format a Jira ticket" tool everyone can use
Agent-Scoped Skill Give one specific agent a specialized capability A specialized tool only one agent needs — like a password reset process that only the IT Agent uses
Personal Skill Create something just for yourself Your personal shortcut — like a custom way you prefer to give feedback or summarize meetings

Framework in Practice

Concrete examples of choosing the right primitive for each use case

When to use an Agent use agent

✘ "Weekly Reporting Skill" — just drafts documents when called
✔ "Weekly Report Agent" — runs every Friday, gathers metrics, drafts report, sends to stakeholders
Why Agent: Needs independent scheduling (triggers), its own service account for data access, lifecycle controls, and version management. A simple writing skill can't handle the orchestration or governance requirements.

When to use a Global Skill use global skill

✘ send_email() as agent-scoped skill inside Sales Agent
✔ send_email() as global skill — reusable by Assistant and all agents
Why Global Skill: Email sending is a horizontal capability everyone needs — Sales, IT, Marketing, HR. Making it global allows the Assistant and multiple agents to leverage it without rebuilding.

When to use an Agent-Scoped Skill use scoped skill

✘ reset_password_with_approval_and_mfa_check() as global skill
✔ reset_password_with_approval_and_mfa_check() as agent-scoped skill inside IT Agent
Why Agent-Scoped Skill: This complex, multi-step operation with approval workflows and MFA verification is specific to IT Helpdesk processes. It's too specialized to be a general-purpose Assistant capability.

When to use an Agent use agent

✘ "Weekly Reporting Skill" — only helps you write when you ask
✔ "Weekly Report Agent" — automatically gathers data every Friday, creates the report, and sends it to your team
Why Agent: This job needs to run on its own schedule, pull data from multiple systems, and deliver results without anyone babysitting it. A skill can help you write — but an agent does the whole job.

When to use a Global Skill use global skill

✘ "Send Email" built only for the Sales team
✔ "Send Email" available to everyone — Sales, IT, Marketing, HR
Why Global Skill: Sending emails is something every team needs. Make it once, share it company-wide, and every team (and every agent) can use it — no need to rebuild the same thing over and over.

When to use an Agent-Scoped Skill use scoped skill

✘ "Password Reset with Approval" available to everyone
✔ "Password Reset with Approval" only available to the IT Help Desk Agent
Why Agent-Scoped Skill: Password resets involve security checks and approval steps that are specific to the IT team. This capability should only be used by the IT Agent — it doesn't make sense as a general tool for everyone.
Agents and Skills are complementary. Build the right primitive for the right job.
