Agents vs. Skills

Key distinctions between Glean's two agentic primitives — and a decision framework for when to use each.

Core Premise

As agentic primitives and core AI fundamentals strengthen, Assistant + Skills will increasingly act like an agent and digital coworker, handling the majority of end-user use cases with growing autonomy and intelligence.

The question is not which product is more agentic. The question is what Glean Agents uniquely provide. Agents are containers that package skills, tools, and knowledge with enterprise-grade governance, scoping, identity, and operational controls for high-value production workflows.
⚖

Durable Differentiators

Architectural properties of agents that skills cannot replicate without becoming agents themselves
1 Independent Identity durable
What Agents Enable
  • Execute with their own service account identity, separate from individual users
  • Access to privileged tools that end-users should never have
  • Background execution without requiring a logged-in user
  • Audit trails that distinguish agent actions from human actions
Why This Matters

Some operations require elevated permissions that users should never possess directly. An IT Admin Agent can reset passwords or provision software because it has privileged access, not because the requesting user does. This separates capability from user permissions.

Example
User asks: "Reset my password"
IT Agent → Executes with IT service account credentials (user never has this permission)
Audit log → Shows "IT Agent" performed the action, not the user
🤖 Agent
Executes with its own service account identity and privileged access
✨ Skill
Always executes in the user's security context, inheriting their permissions only
Why this is durable: Identity separation is a fundamental architectural boundary. A skill that gains its own service account and execution context is, by definition, an agent.
2 Installability & External Surface Deployment durable
What Agents Enable
  • Deployment to external surfaces (Slack channels, ServiceNow, Teams)
  • Embedded experiences within third-party systems
  • Standalone invocation outside the Glean Assistant
Why This Matters

Agents can be installed where users already work. An HR Agent lives in the #hr-help Slack channel. An IT Agent is embedded in the ServiceNow portal. Skills are invoked capabilities within the Assistant — they don't have an independent presence in external systems.

Example
✔ IT Agent — Installed in #it-help Slack channel, responds 24/7
✘ IT Skill — Can only be invoked inside Glean Assistant
Why this is durable: Skills are ingredients consumed by assistants and agents. They don't have an addressable identity or presence outside the assistant context.
3 Scoped & Constrained Execution Environment durable
What Agents Enable
  • Constrained tool access — agents only use tools relevant to their domain
  • Knowledge boundaries — IT agents see only IT docs, not finance data
  • Explicit tool and knowledge manifests per agent
  • Guaranteed guardrails — the agent cannot break out of its scope
Why This Matters

General-purpose assistants have broad access, creating security and relevance problems. An IT Helpdesk Agent should never search through finance documents. Agents enforce these boundaries by design. Skills operate in the user's context with their full access.

🤖 Agent
Scoped to IT documentation, ticketing systems, password reset tools, software provisioning
✨ Skill
Could accidentally surface finance data when helping with an IT question, creating compliance risks
Why this is durable: Skills run within the assistant's broad context. An agent runs in a separate, constrained environment. A skill that gains enforced scoping is effectively becoming an agent.

Near-Term Differentiators

Agent advantages for the next 6–12 months — skills will likely gain parity as the ecosystem matures
4 Observability, Analytics & Feedback Loops near-term
What Agents Enable Today
  • Per-agent execution traces (steps, tools used, latency, errors)
  • Agent-level dashboards (success rate, ticket volume, resolution time)
  • ROI metrics (cost per resolved ticket, time saved per agent)
  • SLA monitoring and alerting
  • Qualitative feedback routing: thumbs up/down routed directly to the agent owner
  • Clear attribution: when something fails, you know exactly which agent failed
  • SME-driven iteration: agent owners receive feedback and iterate based on domain expertise
Example
IT Agent Dashboard: 1,247 password resets • 89% success rate • avg 2.3 min • $12K saved

Feedback loop → User thumbs down VPN response → Routes to Sarah (IT Lead) → She updates prompts → Success rate: 89% → 94%
Why this will erode: As skills gain creators/owners and admin-level distribution, the scaffolding for per-skill observability will be built. The execution boundary argument holds today but weakens as skills become more structured.
5 Cost Optimization & Budget Guardrails near-term
What Agents Enable Today
  • Per-agent model selection and budget caps aligned with ROI
  • Real-time cost tracking per agent
  • Different cost tolerances based on business value
Examples
💵 Cost Reduction
IT Helpdesk Agent
Saves $75K/yr in labor
Uses Haiku ($240/yr)
Net savings: $74,760
📈 Revenue Driver
Sales Agent
Generates $3M in new deals
Uses Opus ($6K/yr)
ROI justifies premium model
Why this will erode: Per-skill model selection and cost tracking are engineering problems, not architectural ones. As skills gain structure and attribution, cost optimization could extend to them.

Converging Capabilities

Agent features that skills must adopt now for enterprise readiness — customers are already asking
6 Governance & Permission Control skills need now
What Agents Have Today
  • Centralized control over who can create, distribute, and manage containers
  • Approval workflows for deployment (compliance, security review, rollback)
  • Deployment gates (gradual rollout, A/B testing of versions)
  • Access control for which users/groups can invoke specific agents
Why Skills Need This Now

In enterprises, not everyone should create production workflows that touch sensitive systems. Skills currently lack organizational control. Skills that include code execution (Python via AFS) are especially high-risk for ungoverned sharing. Customers are already demanding governance for skills.

Example

Your marketing team publishes a "Brand Voice" skill. Today, any user could upload a competing skill with outdated brand guidelines. Without governance: no admin control over active skills, no approval flow, no way to ensure everyone uses the latest version.

What we should build: Reuse the governance primitives already built for agents (admin roles, sharing controls, approval flows) and extend them to skills rather than building parallel systems.
7 Packaging, Versioning & Testing skills need now
What Agents Have Today
  • Versioned, deployable artifacts (Agent v1.2)
  • Independent test suites and regression testing per agent
  • Rollback capabilities when new versions fail
  • Staged deployments (dev → staging → production)
Example
Sales Agent v2.0 changes CRM update logic:

✔ Test v2.0 against regression suite before rollout
✔ Deploy to 10% of users, monitor for errors
✔ Rollback to v1.9 if issues arise
What we should build: GitHub-based skill management. Sync skills from repos, support PRs as the change control mechanism, and provide version history and rollback within Glean admin. We need this Git-based ADLC support for agents as well.

Decision Framework

Agents and skills are complementary — agents are production workflows that often use skills as their tools
Build This When Characteristics
Agent Needs governance, scoping, identity, or operational controls Independent identity, scoped tools/knowledge, governance, versioning, observability, SME ownership, cost optimization
Subagent Needs domain reasoning, no independent lifecycle Domain expertise, multi-step decisions, delegated execution within another agent
Global Skill Simple, reusable capability for many users; deployed by admin or builder Single-purpose, stateless, horizontal use case (brand tone, JIRA formatting) where standardization matters org-wide
Agent-Scoped Skill Specialized capability for one agent's domain Complex, domain-specific logic needed only by that agent
Personal Skill Built or imported by a user for personal use only Same as global skill, but idiosyncratic to the end user's preference (feedback giving style)

Framework in Practice

Concrete examples of choosing the right primitive for each use case

When to use an Agent use agent

✘ "Weekly Reporting Skill" — just drafts documents when called
✔ "Weekly Report Agent" — runs every Friday, gathers metrics, drafts report, sends to stakeholders
Why Agent: Needs independent scheduling (triggers), its own service account for data access, ADLC controls, and version control. A simple writing skill can't handle the orchestration, lifecycle, or governance requirements.

When to use a Global Skill use global skill

✘ send_email() as agent-scoped skill inside Sales Agent
✔ send_email() as global skill — reusable by Assistant and all agents
Why Global Skill: Email sending is a horizontal capability everyone needs — Sales, IT, Marketing, HR. Making it global allows the Assistant and multiple agents to leverage it without rebuilding.

When to use an Agent-Scoped Skill use scoped skill

✘ reset_password_with_approval_and_mfa_check() as global skill
✔ reset_password_with_approval_and_mfa_check() as agent-scoped skill inside IT Agent
Why Agent-Scoped Skill: This complex, multi-step operation with approval workflows and MFA verification is specific to IT Helpdesk processes. It's too specialized to be a general-purpose Assistant capability.
Agents and Skills are complementary. Build the right primitive for the right job.
